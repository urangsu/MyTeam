# 🚀 MyTeam 프로젝트: 신규 개발자를 위한 가이드 및 아키텍처 개요

반갑습니다! **MyTeam**은 인공지능 에이전트들이 단순한 챗봇을 넘어, 하나의 '팀'으로서 사용자(리더)와 협업하는 macOS 데스크탑 어플리케이션입니다. 이 문서는 현재의 구조를 빠르게 파악하고, 앞으로 어떤 부분을 개선해 나가야 할지 가이드를 제공합니다.

---

## 1. 어플리케이션 소개 (What is MyTeam?)

MyTeam은 사용자의 데스크탑 위에 떠 있는 **'지능형 부유식 패널(Floating Panels)'** 기반의 AI 비서 팀입니다. 

- **멀티 에이전트 협업**: 전략가, 디자이너, 개발자 등 각기 다른 페르소나를 가진 에이전트들이 서로 대화하며 문제를 해결합니다. (AutoGen 스타일의 그룹 채팅)
- **온디바이스 보이스 크로닝**: Microsoft ONNX Runtime을 사용하여 클라우드 연결 없이도 캐릭터 고유의 목소리로 대화합니다.
- **Context-Aware Memory**: 사용자와의 대화 중 핵심 사실(Key Facts)을 추출하여 기억하고, 다음 대화에 반영합니다.

---

## 2. 핵심 아키텍처 구조 (Core Framework)

MyTeam은 **SwiftUI + AppKit + Swift 6 Concurrency**를 기반으로 설계되었으며, 주요 로직은 `shared` 싱글톤 패턴을 따르는 매니저 객체들이 담당합니다.

### 2.1 UI 및 윈도우 관리 (`AgentWindowManager`)
- **역할**: macOS의 `NSPanel`을 확장한 `FloatingPanel`을 관리합니다.
- **특징**: SwiftUI 뷰를 AppKit 윈도우에 주입(Hosting)하여, 데스크탑 어디든 자유롭게 떠다니는 UI를 구현합니다. 에이전트의 감정 상태와 입력 인디케이터를 전역적으로 추적합니다.

### 2.2 지능형 오케스트레이션 (`TeamOrchestrator`)
- **역할**: 대화의 "두뇌" 역할을 하며, 다음에 어떤 에이전트가 말해야 할지 LLM을 통해 결정합니다.
- **패턴**: `IntentRouter`를 통해 사용자의 요청이 '업무(Task)'인지 '수다(Chitchat)'인지 구분하고 적절한 에이전트를 소환합니다.

### 2.3 음성 합성 파이프라인 (`SpeechManager` & `OnDeviceTTSManager`)
- **SpeechManager**: Apple의 고유 TTS와 온디바이스 TTS를 중개하는 컨트롤러입니다.
- **OnDeviceTTSManager**: Chatterbox(Zero-shot TTS) 모델을 ONNX 환경에서 실구동합니다. 음소 추출(BPE), 텍스트 임베딩, 자기회귀(AR) 디코딩, 파형 합성의 4단계 파이프라인으로 구성됩니다.

### 2.4 AI 서비스 추상화 (`AIService`)
- **역할**: OpenAI(GPT), Google(Gemini), Anthropic(Claude) API를 통합 관리합니다.
- **특징**: 특정 서비스 불능 시 자동으로 다른 모델로 전환(Fallback)하는 유연한 구조를 갖추고 있습니다.

## 3. 신규 개발자가 개선해야 할 과제 (Improvement Roadmap)

현재 빌드 안정성과 동시성 문제는 상당 부분 해결되었으나, 더 높은 수준의 고도화를 위해 아래의 개선 작업들이 필요합니다.

### 🟢 1. 현대적인 상태 관리 (`Observation` 프레임워크 도입)
- **현재**: `ObservableObject`와 `Combine`을 널리 사용 중입니다.
- **개선**: iOS 17/macOS 14 이상에서 권장되는 `@Observable` 매크로로 전환하여 뷰 업데이트 성능을 최적화하고 코드를 간소화해야 합니다.

### 🔵 2. 의존성 주입 (Dependency Injection) 패턴 도입
- **현재**: 거의 모든 컴포넌트가 `.shared` 싱글톤에 의존하고 있습니다.
- **개선**: 테스트 용이성(Testability)을 위해 싱글톤을 배제하고, `Environment` 객체나 가벼운 DI 컨테이너를 통해 인스턴스를 주입받는 방식으로 변경이 권장됩니다.

### 🟡 3. 리소스 관리 최적화 (ONNX Lazy Loading)
- **현재**: 앱 시작 시 모든 ONNX 모델과 참조 오디오를 메모리에 적재합니다.
- **개선**: 에이전트가 실제로 사용될 때 모델을 로드하거나, 사용하지 않는 모델을 메모리에서 해제하는 캐싱 전략이 필요합니다.

### 🔴 4. 테스트 코드(XCTest/Swift Testing) 작성
- **현재**: 로직 검증이 주로 실구동 테스트에 의존하고 있습니다.
- **개선**: 특히 `TeamOrchestrator`의 발언자 선택 로직이나 `TextSanitizer`의 정제 로직에 대해 단위 테스트(Unit Test)를 구축해야 합니다.

### 🟣 5. 페르소나 데이터 분리 (Separation of Concerns)
- **현재**: 에이전트의 성격(Prompt)과 설정 데이터가 코드 내에 하드코딩된 부분이 많습니다.
- **개선**: YAML이나 JSON 설정 파일로 외부화하여, 코드 수정 없이도 새로운 에이전트를 추가하거나 성격을 조정할 수 있는 데이터 주도형(Data-driven) 구조로 발전시켜야 합니다.
